# Deployment — Render Configuration and Rules

## Purpose

This file defines how deployment to QA and PROD environments works on Render. Read `docs/rules/constraints.md` before this file. Any conflict resolves in favor of `constraints.md`.

---

## Overview

Deployment is handled by Render.com. Two independent services are configured — one for QA and one for PROD. Both services pull Docker images from GitHub Container Registry (ghcr.io). Render does not build anything — it only runs the image it is given.

The pipeline triggers each deploy via an HTTP POST to a Render deploy hook URL. Render then pulls the image tagged with the current commit SHA and starts the new container. Once the new container is healthy, the previous one is terminated. This is zero-downtime deployment.

## Service configuration

| Aspect | QA | PROD |
|--------|----|------|
| Service name | `forum-app-qa` | `forum-app-prod` |
| Image source | `ghcr.io/carpinetioctavio/forum-app-cloud-deploy-backend` | Same |
| Image tag | Commit SHA | Same commit SHA as QA |
| Deploy trigger | Automatic (pipeline stage 7) | Manual approval required (pipeline stage 9) |
| Environment | GitHub environment `qa` | GitHub environment `prod` |
| Runtime config | QA environment variables | PROD environment variables |

## Environment variables per service

Environment-specific values are configured directly in each Render service. They are never stored in source code, Dockerfiles, or pipeline YAML files.

**Code must analyze the existing backend source to determine the full list of required environment variables before configuring either service.** Report the findings before any Render configuration is documented or applied.

Known variables (to be confirmed by Code after analysis):

## Secrets

The following secrets must be configured in GitHub before the pipeline can deploy:

| Secret name | Scope | Value |
|-------------|-------|-------|
| `RENDER_DEPLOY_HOOK_QA` | GitHub environment `qa` | Deploy hook URL from Render QA service |
| `RENDER_DEPLOY_HOOK_PROD` | GitHub environment `prod` | Deploy hook URL from Render PROD service |

Deploy hook URLs are generated by Render per service. They are treated as credentials — never logged, never committed, never exposed in pipeline output.

---

## Image promotion — same artifact, both environments

The image deployed to PROD is the same image that was deployed to QA, identified by the same commit SHA tag. Render does not pull a new image for PROD — it pulls the same tag that passed QA validation.

This is the technical guarantee of environment parity: the artifact is identical. The only differences between QA and PROD are external configuration values injected at runtime.

---

## Rollback procedure

If a PROD deployment causes a failure:

1. Identify the last known good commit SHA from the GitHub Actions run history.
2. In the Render PROD service, update the image tag to the last known good SHA.
3. Trigger a manual deploy in Render.
4. The previous image — which is still available in ghcr.io — is pulled and started.

Rollback does not require rebuilding anything. The image is already in the registry. This is only possible because images are tagged with commit SHA and never overwritten.

---

## QA vs PROD — legitimate differences

The following differences between QA and PROD are expected and acceptable:

- Runtime environment variables (database path, API URLs, credentials)
- Render service resource allocation (CPU, memory — constrained by free tier)
- Deploy trigger (automatic vs. manual approval)

The following must never differ between QA and PROD:

- The Docker image (same artifact, same SHA)
- The application code
- The Dockerfile used to build the image

## What Code must do before configuring or documenting deployment

1. Analyze the existing backend source to determine all required runtime environment variables.
2. Analyze the existing frontend source to determine how the backend URL is configured and whether `REACT_APP_API_URL` or an equivalent variable is already in use.
3. Resolve the SQLite persistence pending item documented in `docs/rules/constraints.md` — determine whether a volume, an external database, or another strategy is appropriate for Render's free tier, and report the recommendation before any deployment configuration is finalized.
4. Report all findings before writing or modifying any deployment-related file.
